## v0.4.1：HTTP 429 限流修复与无空白降级

v0.4.0 在首次加载时会同时或连续请求多个 SimCompanies / SimcoTools 数据源。干净安装、频繁重载扩展、同时运行其他工具，或服务端已经处于冷却状态时，这种请求突发可能触发 `HTTP 429`，并使整个面板停在“加载失败”。

v0.4.1 将数据层改为限流安全模式：

- 在 `document_start` 尽早观察游戏页面本来就会加载的只读响应；
- 优先复用页面已经取得的账号、ticker、天气、事件和订单簿数据，不重复请求；
- 所有扩展主动请求按主机串行排队，同一缓存键的并发请求自动合并；
- SimCompanies API 主动请求使用独立五分钟安全门；静态游戏资源不受该门影响；
- 429 有 `Retry-After` 时严格采用服务端时间；没有时默认冷却五分钟，不再做秒级快速重试；
- 已有旧缓存时，429 会立即退回旧快照并标明数据年龄；
- 账号管理费、高管、休闲、天气和生产事件改为“页面观察优先、缓存其次、中性假设最后”，不再在启动时主动形成一组认证请求；
- 干净安装时即使账号接口 429，也会以明确标记的中性账号参数继续尝试加载市场，而不是直接空白；
- 无可用资源/市场缓存时，面板显示来源、冷却倒计时，并只自动重试一次；不会无限循环；
- 修复了早期页面桥接响应可能在监听器安装前被发送并丢失的问题。

### 429 时现在会发生什么

```text
有可用缓存
→ 立即显示结果
→ 状态栏警告当前使用旧快照

没有可用缓存
→ 立即停止继续请求
→ 显示具体主机、接口和冷却倒计时
→ 冷却结束后自动重试一次
→ 再失败则等待用户手动刷新
```

账号参数无法完整读取时，所有结果的可信度会降为“估算”，状态栏会列出缺失字段。不要把中性参数结果当成你的最终账号利润；等游戏页面自然加载相应数据后再刷新。


## v0.4.0：多级产业链优化

v0.4.0 新增两条复杂路线：

1. **最优产业链 → 交易所**
2. **最优产业链 → 零售**

它不再把每种成品都假设为“所有原料直接从交易所采购”。插件会递归展开完整配方图，并对每个中间品比较：

```text
从交易所外购
vs
自己生产，并继续优化它的上游
```

最终输出一个平衡的 make-or-buy 方案，包括：

- 每个自产节点的小时需求、每级产能、所需建筑级数和工资；
- 每个外购节点的小时需求、当前价格和小时现金成本；
- 总生产建筑级数；
- 末端产量或零售销量；
- 整条链的小时利润和 PPHPL；
- 产业链深度、自产节点数、外购节点数和价格来源。

### 为什么不能只递归比较“自产单位成本”

最低单位现金成本不一定等于最高 PPHPL。一个中间品可能自产更便宜，但需要大量低效建筑级；直接外购虽然贵一点，却可能释放大量建筑产能，使整条链的每级利润更高。

v0.4.0 使用标准的分式规划迭代。对候选 PPHPL `λ`，每个上游节点比较：

```text
外购增广成本 = 市场价格

自产增广成本
= 自产现金成本
+ λ × 自产占用建筑级数
+ 所有上游的最优增广成本
```

然后不断更新：

```text
λ = 整条链净利润 ÷ 整条链总建筑级数
```

在当前线性模型下，这会得到全局 make-or-buy 最优解，而不是逐层贪心地选择最低单位成本。

## 五条可比较路径

### 1. 生产 → 交易所

末端商品自己生产；所有直接原料按交易所价格采购。

### 2. 交易所购入 → 零售

成品直接从交易所采购，搜索小时利润最高的零售价。

### 3. 生产 → 零售

直接原料从交易所采购，成品自己生产并零售；同时计入生产建筑和零售建筑占用。

### 4. 最优产业链 → 交易所

末端成品必须自产。插件递归决定每个上游节点是自产还是外购，并优化：

```text
PPHPL
=（末端交易所净收入 − 全链现金成本）
 ÷ 全链生产建筑级数
```

交易所净收入计入末端手续费、末端运输和衰减。内部自产中间品不会重复收取交易所手续费或末端运输。

结果按“**1 个末端生产建筑级**”归一化，因此上游所需级数通常是小数。

### 5. 最优产业链 → 零售

插件对每个候选零售价同时优化：

- 当前零售速度；
- 末端零售收入和商店工资；
- 为满足该零售速度所需的完整生产链；
- 每个上游节点的自产/外购决策；
- 1 个零售建筑级加全部上游生产级数后的总 PPHPL。

```text
PPHPL
=（零售收入 − 商店工资 − 全链现金成本）
 ÷（1 个零售级 + 平衡上游生产级数）
```

结果按“**1 个末端零售建筑级及其平衡上游**”归一化。

## 产业链策略

在“计算范围、季节口径与个人参数”中可以选择：

### 全局最优 make-or-buy（推荐）

每个中间品都由优化器决定自产或外购。若最终最优解没有选择任何上游自产，该复杂路线会被省略，因为它与普通“生产 → 交易所/零售”路线等价；诊断区会显示“没有上游整合收益”。

### 可生产上游全部自产

只要某个上游资源能够生产，就优先自产；只有不可生产、递归深度超限或没有可解配方时才尝试外购。适合评估完全垂直整合，但不保证 PPHPL 最高。

### 产业链最大深度

默认 12，范围 1–30。用于防止异常 API 数据中的循环或过深依赖。正常 SimCompanies 产业链通常不需要调高。

## 性能设计

复杂产业链默认关闭，避免每次普通查看都产生额外计算。

启用后：

- “产业链 → 交易所”会遍历所有具备上游配方的生产资源；
- “产业链 → 零售”只遍历传统零售商品；
- 同一个末端商品的交易所和零售路线共享递归策略缓存；
- 每个优化器只保留有限数量的最近 `λ` 状态，避免内存持续增长；
- UI 会先显示“正在优化产业链”，再开始重计算。

在本地 151 资源合成压力测试中：

- 全资源产业链 → 交易所约 0.3 秒；
- 全资源交易所路线加 50 个零售末端的产业链零售约 2.5 秒。

实际时间取决于浏览器、开启路线、配方深度和零售候选数量。

## 如何使用产业链优化

1. 保持 Q0；Q0 的配方与市场价格口径最可靠。
2. 打开“计算范围、季节口径与个人参数”。
3. 勾选“最优产业链 → 交易所”和/或“最优产业链 → 零售”。
4. 策略选择“全局最优 make-or-buy”。
5. 等待状态栏完成递归优化。
6. 在路径筛选中选择产业链路线。
7. 点击结果行展开“平衡产业链方案”。
8. 对排名靠前的末端产品和关键外购原料逐一打开交易所页面，让插件捕获真实最低卖单。

展开结果后重点核对：

- 总产业链级数；
- 自产和外购节点；
- 最大上游深度；
- 外购原料价格来源；
- 末端销量/产量；
- 小数级数是否能被你的真实建筑布局近似实现；
- 关键外购原料是否有足够订单簿深度。

## 产业链模型包含什么

每个自产节点均使用当前账号和资源状态：

- 当前 `producedAnHour`；
- 可核对的官方生产事件；
- Pumpkin 等年度产季当前速度；
- Tree 质量时长和研究速度；
- 矿藏丰度；
- 个人生产加成；
- 游戏加速；
- 当前管理费和 COO；
- 机器人 3% 减薪设置；
- 当前质量的配方原料数量。

每个外购节点使用当前价格优先级：

```text
已访问交易所页面的最低卖单
→ SimCompanies 最低价 ticker
→ SimcoTools VWAP 回退
```

季节零售、Summer 天气、淡季筛选和冰淇淋衰减继续沿用 v0.3.0 的机制。

## 产业链模型暂不包含什么

### 1. 建筑和升级资本

当前优化目标是运营 PPHPL，不包含建造/升级材料、建设等待时间和回收损失。

### 2. 整数建筑布局

模型允许小数建筑级，用来表示稳态产能配比。真实建筑只能是整数级，因此结果是理论上限。实际部署应将关键节点向上取整，再重新检查瓶颈。

### 3. 新增建筑对全公司管理费的二次影响

链上每个节点均使用你账号当前有效管理费倍率。模型没有模拟新增若干建筑级后，整个公司的管理费进一步上升，也没有计入这对现有业务工资的外部性。

### 4. 订单簿深度和价格冲击

最低卖价或 VWAP被视为恒定单位价格。大量采购会吃掉多档订单，实际成本可能更高。

### 5. 库存、启动资金和换线

没有模拟初始库存、现金占用、生产批次、停机、换线、合同到货节奏和仓库安全库存。

### 6. 多产品组合

v0.4.0 优化的是“单个末端产品的一条平衡产业链”。它不是在固定预算、建筑槽位和时间范围下，同时选择多个末端产品的投资组合。后者需要整数规划或混合整数线性规划。

### 7. Q1 以上质量组合

除 Tree 外，Q1 以上仍沿用同质量原料价格近似。建议把复杂产业链排名主要用于 Q0。

## 季节机制摘要

v0.3.0 已实现并保留：

- 当前资源产量与官方生产事件核对，避免修正双算；
- Pumpkin 当前年度产季速度；
- 2026 Ramadan、Easter、Summer、Halloween 和 X-mas 窗口；
- 当前可执行与淡季规划筛选；
- Summer 实时天气；
- Chocolate icecream / Apple icecream 5%/小时复合衰减；
- Tree Q0–Q12 时长/原料倍增和 4%/研究质量级速度加成。

内部自产中间品按稳态即时流转处理，不额外施加仓库存放时间；末端衰减商品仍使用“衰减持有 / 挂单小时”设置。

## 安装或升级

1. 解压 ZIP。
2. 打开 `chrome://extensions`；Edge 使用 `edge://extensions`。
3. 开启“开发者模式”。
4. 首次安装：点击“加载已解压的扩展程序”，选择 `simco-profit-radar` 文件夹。
5. 升级：用新文件覆盖旧扩展目录，在扩展管理页点击“重新加载”。
6. 刷新 SimCompanies；扩展也会尝试自动刷新已打开且加载完成的游戏标签页。
7. 点击右下角“利润雷达”，标题应显示 `v0.4.1`。

旧设置会自动迁移。两条产业链路线默认关闭，产业链策略默认“全局最优”，最大深度默认 12。

## 默认行为

- 默认质量：Q0。
- 默认矿藏丰度：90%。
- 默认零售批量：1,000。
- 默认最多显示：500 条。
- 原三条路线默认开启。
- 两条复杂产业链路线默认关闭。
- 默认隐藏负利润。
- 交易所手续费默认按 4% 计算。
- 市场、账号、零售、天气、事件和当前产能缓存 5 分钟。
- 已访问订单簿最低价覆盖 15 分钟后失效。

## 数据与安全

扩展仅使用只读 GET 数据，不自动买卖、生产或提交任何游戏操作。业务代码不包含 POST、PUT、PATCH 或 DELETE 请求。

所有计算均在本地浏览器完成。设置和缓存保存在 `chrome.storage.local`；没有遥测、广告或分析代码。

普通权限只有 `storage`。主机权限限定为 SimCompanies 和 SimcoTools 数据域名。
